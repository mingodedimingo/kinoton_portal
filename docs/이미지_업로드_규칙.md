# 🖼️ 이미지/파일 업로드 — 절대 규칙 (재발 방지 가이드)

> **이 문서를 먼저 읽지 않고 업로드/스토리지 코드를 수정하지 마세요.**
> 게시판 이미지가 "깨진 아이콘"으로만 보이는 버그가 **최소 6번 재발**했습니다.
> 매번 다른 "근본 원인"을 고쳤지만, 아래 **불변식(invariant)** 을 어기면 또 깨집니다.
> 대상 파일: `server/_core/index.ts`, `server/storage.ts`, `server/_core/storageProxy.ts`,
> `client/src/components/RichEditor.tsx`, `client/src/pages/BoardDetailPage.tsx`

---

## 0. TL;DR — 한 문장 규칙

**업로드 엔드포인트는 `storagePut()`이 돌려주는 상대경로 `/api/img/{key}` 만 반환·저장한다.
절대 URL도, presigned/signed URL도, base64 data URI도 클라이언트에 주거나 DB에 저장하지 않는다.**

이미지를 실제로 보여주는 것은 `storageProxy`의 책임이다 (서버에서 파일을 받아 그대로 흘려보냄).

> ### ⛔ 서빙 경로는 반드시 `/api/` 하위여야 한다 (가장 중요한 교훈)
>
> 예전에는 `/manus-storage/{key}` 경로로 서빙했다. **이것이 6번 재발의 진짜 원인이었다.**
> 배포 환경에서 `/manus-storage/*` 요청은 **Manus/Cloudflare 플랫폼 엣지가 가로채
> Express 앱(Cloud Run 컨테이너)에 도달조차 못 한다.** 그래서 앱이 등록한
> `app.get("/manus-storage/*")` storageProxy는 프로덕션에서 절대 실행되지 않고,
> 엣지가 9바이트 `Not found`(404)만 돌려준다 → 모든 이미지가 깨진 아이콘.
>
> **진단법:** 깨진 이미지 URL을 `fetch`해서 응답 헤더를 보라.
> `x-powered-by: Express`와 `x-cloud-trace-context`가 **있으면** 앱에 도달한 것(코드 문제),
> **없으면** 엣지가 가로챈 것(경로 문제 → `/api/` 하위로 옮겨야 함).
> `/api/*` 경로는 컨테이너에 도달함이 검증됐다. 그래서 서빙을 `/api/img/*`로 옮겼다.

## 1. 올바른 데이터 흐름 (이대로만 두면 안 깨진다)

```
[업로드]
 브라우저 ──multipart/form-data──> POST /api/upload-image (또는 /upload-file)
   → storagePut(key, buffer, mime)        // 내부에서 해시 접미사 추가 후 S3에 PUT
   → 반환된 { key, url } 에서 url 사용     // url === "/api/img/{실제key}"
   → res.json({ url, ... })                // ✅ 상대경로만 반환

[저장]
 게시글 본문 HTML / 첨부 JSON 에 "/api/img/{key}" 상대경로만 저장
   (절대 도메인·만료 토큰이 본문에 박히지 않음 → 영구히 유효)

[서빙/렌더]
 <img src="/api/img/{key}">  (same-origin)
   → GET /api/img/{key}  →  storageProxy
   → forge presign/get 으로 signed URL 획득 (요청마다 새로)
   → signed URL을 "서버에서" fetch 해서 바이트를 res.end(buffer) 로 파이프  // ✅ 브라우저는 signed URL을 보지 못함
```

**핵심:** signed URL은 **서버 내부에서만** 쓰고 즉시 소비한다. 브라우저로 절대 내보내지 않는다.

## 2. ✅ DO / ❌ DON'T

### ❌ 하지 말 것 (전부 과거에 깨뜨린 실제 사례)

| 금지 | 왜 깨지나 |
|------|-----------|
| ❌ 업로드 응답에 `storageGetSignedUrl()` 결과(presigned URL) 반환 | 브라우저 직접 접근 시 **403**, 게시글에 박힌 뒤 **만료**되면 전부 깨짐 |
| ❌ 절대 URL 저장 (`${origin}/...`, `http(s)://...run.app/...`, CloudFront 도메인) | 환경(프리뷰/배포)마다 도메인이 달라 불일치, 만료 토큰 포함 |
| ❌ `storagePut`에 **넘긴 입력 key**로 URL 재구성 | `storagePut`이 해시 접미사를 붙이므로 실제 저장 key와 불일치 → 404/403 |
| ❌ 본문 이미지를 base64 data URI / 대용량 JSON(tRPC)로 전송 | Cloudflare WAF가 244KB+ JSON을 **error 1010**으로 차단 |
| ❌ `BoardDetailPage`의 DOMPurify `ALLOWED_TAGS`/`ALLOWED_ATTR`에서 `img`/`src` 제거 | 렌더 단계에서 `<img>`가 통째로 제거됨 |
| ❌ `storageProxy`를 307 리다이렉트 방식으로 되돌리기 | 브라우저가 리다이렉트된 signed URL에 직접 접근 → 403 |
| ❌ 서빙 경로를 `/manus-storage/*`(또는 `/api/` 밖)로 되돌리기 | 배포 시 플랫폼 엣지가 가로채 앱에 도달 못 함 → **항상 404**, 이미지 깨짐 |

### ✅ 할 것

- ✅ 업로드 응답 `url` = `storagePut(...)`가 반환한 `url` (상대 `/api/img/{key}`)
- ✅ 본문·첨부에는 **상대경로만** 저장
- ✅ 서빙은 `storageProxy`가 **서버에서 파이프**(`res.end(buffer)`)
- ✅ DOMPurify에서 `img` 태그와 `src` 속성 허용 유지
- ✅ 확장자 기반 MIME 매핑 유지 (multer가 `application/octet-stream`을 줄 때 대비)
- ✅ 수정 후 반드시 `pnpm test` 통과 확인 (계약 테스트가 위반을 잡음 → §4)

## 3. 🪦 실패한 시도들의 무덤 (같은 길을 또 가지 마세요)

> 아래는 전부 "이번엔 진짜 근본 원인"이라며 커밋됐지만 **재발**한 시도들입니다.
> 증상이 같다고 같은 방향(절대 URL / presigned / base64)으로 다시 가지 마세요.

1. **절대 URL(origin 기반) 반환** → 프리뷰/배포 도메인 불일치로 깨짐. (커밋 `ff39ff1`, `ea9e299`)
2. **base64 + tRPC mutation 전송** → Cloudflare WAF 1010 차단. (커밋 `f4fdfa6`, `d8aa95b`)
3. **키 불일치 (해시 접미사 무시)** → `storagePut` 반환 key를 안 쓰고 입력 key로 URL 생성 → 403 AccessDenied. (수정 `e0ee7ef`)
4. **DOMPurify가 `<img>` 제거** → 본문에서 이미지 태그 사라짐. (수정 `4d3d2f6`, `a5b24bd`)
5. **presigned URL 직접 반환** → 브라우저 403 + 만료. (회귀 `f6d2f9b` → PR #1에서 수정)
6. **`/manus-storage/*` 경로로 서빙 (← 6번 재발의 진짜 원인)** → 코드(`storageProxy`)는 옳았지만
   배포 환경에서 `/manus-storage/*`는 **플랫폼 엣지가 가로채 앱에 도달조차 못 함**. 로컬(`pnpm dev`)엔
   엣지가 없어 200이 떠서 "고쳤다"고 착각했고, 배포하면 항상 404 → 깨짐. **로컬에서만 검증한 것이 함정.**
   서빙을 컨테이너에 도달하는 `/api/img/*`로 옮겨 해결. (헤더에 `x-powered-by`/`x-cloud-trace` 유무로 진단)

**교훈:** 깨짐의 원인은 (a) 클라이언트에 **불안정한 URL**(절대/만료/불일치)을 줬거나,
(b) 서빙 경로가 **앱에 도달하지 못하는 플랫폼 예약 경로**(`/manus-storage`)였기 때문.
해법은 늘 동일하다 → **안정적인 상대경로 `/api/img/{key}`(반드시 `/api/` 하위) + 서버 파이프 프록시.**
그리고 **반드시 배포(Publish) 후 실제로 확인**할 것 — 로컬 성공은 증거가 아니다.

## 4. 🔒 회귀 테스트 (CI 가드)

`server/upload.contract.test.ts`가 다음 불변식을 검증합니다. `pnpm test`에서 실행됩니다.

- `server/_core/index.ts`의 업로드 엔드포인트가 **presigned URL을 클라이언트로 반환하지 않음**
  (`storageGetSignedUrl`을 이 파일에서 사용하지 않음)
- `storageProxy.ts`가 여전히 **파일을 직접 파이프**(`res.end`)함 (307 리다이렉트로 회귀하지 않음)

테스트가 실패하면 → §2 DON'T를 어긴 것입니다. URL을 상대경로로 되돌리세요.
