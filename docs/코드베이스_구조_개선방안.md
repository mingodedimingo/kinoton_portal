# Kinoton Portal 코드베이스 구조적 문제점 분석 및 개선 방안

## 1. 개요
최근 기능 추가 및 수정 작업(예: 공지사항 카테고리명 변경, 전자결재 링크 수정, 배너 기능 확장 등)을 진행하면서, 단순한 요구사항임에도 불구하고 여러 파일에 걸쳐 수정이 필요하고 연결 고리를 파악하는 데 오랜 시간이 소요되는 문제가 반복적으로 발생했습니다. 

본 문서는 이러한 현상의 근본적인 원인을 코드베이스 구조 측면에서 분석하고, 향후 개발 생산성과 유지보수성을 높이기 위한 선제적 조치 방안을 제안합니다.

## 2. 구조적 문제점 분석

### 2.1. 매직 스트링(Magic String) 및 하드코딩의 산재
가장 큰 문제는 도메인 지식이나 UI 라벨, 외부 링크 URL 등이 한 곳에서 관리되지 않고 여러 컴포넌트에 분산되어 하드코딩되어 있다는 점입니다.

* **공지사항 카테고리 라벨 문제**: '부서'라는 라벨을 '대표이사'로 변경하기 위해 `Home.tsx`, `NoticesPage.tsx`, `NoticeDetailPage.tsx`, `AdminNoticesPage.tsx` 등 최소 4개 이상의 파일을 수정해야 했습니다. `categoryLabel` 매핑 로직이 컴포넌트마다 중복으로 작성되어 있었습니다.
* **외부 시스템 링크 분산**: 전자결재, ERP, 영업시스템, 사내 이메일 등의 URL이 `FullMenuOverlay.tsx`, `PortalLayout.tsx`, `Home.tsx`, `MailPage.tsx` 등에 개별적으로 하드코딩되어 있어, URL이 변경될 때마다 모든 파일을 찾아 수정해야 하는 위험이 있습니다.

### 2.2. 과도하게 결합된 라우팅과 비즈니스 로직
tRPC 라우터(`server/routers.ts`)와 DB 접근 로직(`server/db.ts`) 파일이 너무 방대해졌습니다.

* `routers.ts`는 약 1,200라인에 달하며 모든 도메인(auth, employees, leave, attendance, notices, board, calendar, reservations, banners 등)의 라우터가 하나의 파일에 모여 있습니다.
* `db.ts` 역시 700라인 이상으로 모든 테이블의 CRUD 로직이 집중되어 있어, 특정 도메인(예: 배너)을 수정할 때 다른 도메인 로직과 섞여 있어 파악이 어렵습니다.

### 2.3. 프론트엔드 상태 관리 및 타입 정의의 파편화
프론트엔드에서 사용하는 타입과 백엔드(tRPC)에서 기대하는 타입이 `schema.ts`에 의존하고 있으나, UI에서만 사용하는 파생 상태(예: 폼 입력 상태, 탭 상태)의 타입 정의가 각 페이지 컴포넌트 상단에 개별적으로 선언되어 있습니다. 이로 인해 DB 스키마가 변경될 때 연관된 프론트엔드 컴포넌트들을 추적하기 어렵습니다.

## 3. 선제적 개선 방안 (미리 취해야 할 조치)

이러한 문제들을 해결하고 향후 기능 확장을 원활하게 하기 위해 다음과 같은 리팩토링 및 구조 개선을 제안합니다.

### 3.1. 전역 상수 및 설정 중앙화 (Constant Centralization)
UI 라벨, 외부 링크, 카테고리 매핑 등 변동 가능성이 있는 값들을 중앙에서 관리해야 합니다.

* **조치 방안**: `client/src/config/` 디렉토리를 신설하여 다음과 같이 상수들을 분리합니다.
  * `navigation.ts`: GNB, 퀵메뉴, 전체 메뉴 등에 사용되는 메뉴 구조와 URL(내/외부 링크 포함)을 객체 배열로 정의
  * `categories.ts`: 공지사항, 게시판 등의 카테고리 키-라벨 매핑 객체 정의 (예: `export const NOTICE_CATEGORIES = { all: "전체", company: "회사", dept: "대표이사" }`)
* **기대 효과**: 전자결재 링크나 카테고리명이 변경되더라도 해당 설정 파일 단 한 곳만 수정하면 전체 시스템에 반영됩니다.

### 3.2. 백엔드 로직의 도메인별 모듈화 (Domain-Driven Structure)
비대해진 라우터와 DB 함수들을 도메인별로 분리해야 합니다.

* **조치 방안**: 
  * `server/routers.ts`를 `server/routers/` 디렉토리로 분리하고, `auth.ts`, `board.ts`, `reservation.ts`, `banner.ts` 등으로 나눕니다.
  * `server/db.ts` 역시 `server/repositories/` 하위에 도메인별로 분리합니다.
* **기대 효과**: 배너 기능을 수정할 때는 `banner.ts`만 확인하면 되므로 연결 고리 파악이 훨씬 직관적이고 빨라집니다.

### 3.3. 공통 UI 컴포넌트 추출 (Component Extraction)
여러 페이지에서 반복적으로 사용되는 UI 패턴(예: 카테고리 탭, 롤링 배너, 임직원 프로필 카드)을 독립된 컴포넌트로 추출해야 합니다.

* **조치 방안**: 
  * 조직도, 홈 화면, 출퇴근 관리 등에서 반복 사용되는 임직원 프로필 카드를 `components/EmployeeProfileCard.tsx`로 공통화합니다.
  * 카테고리 선택 탭을 `components/CategoryTabs.tsx`로 공통화하여 `categories.ts`의 설정을 주입받도록 합니다.

## 4. 실행 계획

이러한 개선은 한 번에 진행하기보다, 기능 개발과 병행하여 점진적으로 적용하는 것이 안전합니다.

1. **Phase 1 (즉시 적용 가능)**: `navigation.ts`와 `categories.ts`를 생성하여 하드코딩된 문자열과 링크들을 우선적으로 중앙화합니다.
2. **Phase 2**: tRPC 라우터(`routers.ts`)를 도메인별 파일로 분할합니다.
3. **Phase 3**: DB 접근 로직(`db.ts`)을 레포지토리 패턴으로 분할합니다.
